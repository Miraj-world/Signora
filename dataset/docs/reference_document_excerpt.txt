Voice-of-Customer Product IntelligenceWhat you are buildingA Voice-of-Customer, or VoC, system collects customer feedback from different channels and helps product teams answer:What problems are customers experiencing, which groups are affected, how important are the problems, and what evidence supports working on them?The product is not merely:A review summarizerA sentiment-analysis dashboardA chatbot over support ticketsAn automatic feature-prioritization machineA stronger system converts thousands of scattered customer statements into evidence-backed product opportunities.For example, instead of returning:Many customers dislike onboarding.It should return:Enterprise administrators struggle with onboarding because SSO configuration and bulk user imports require support assistance. The issue appeared in 43 support tickets, 11 sales conversations, and six churn interviews during the last 90 days. It was associated with four delayed deals. However, the problem was rarely mentioned by companies with fewer than 50 employees.Every conclusion should link back to the original feedback.1. The underlying business problemProduct teams receive information through:Support ticketsSales callsApp-store reviewsCustomer interviewsNPS and CSAT responsesFeature-request formsChurn surveysCommunity discussionsGitHub issuesAccount-manager notesSocial mediaProduct usage eventsThese sources usually remain disconnected.A support team may know customers are struggling with exports. Sales may know the missing export functionality is blocking deals. Product managers may see a feature request but not know its revenue implications.Your system connects them:Support ticket    +Sales objection    +Customer review    +Churn explanation    +Product usage data    ↓One evidence-backed product problemExisting platforms already compete here. Productboard Pulse centralizes feedback, detects topics, supports feedback analysis, and connects feedback with feature ideas. Dovetail analyzes inputs such as support tickets, app reviews, product feedback, and survey responses. Enterpret combines feedback from support, sales, product analytics, app stores, and social channels. Chattermill similarly analyzes reviews, surveys, support conversations, calls, and social data. (productboard.com)Therefore, you should not try to build a complete Productboard competitor. Build a focused, technically rigorous portfolio system.2. The exact product I recommendWorking concept: SignalLensA product-intelligence platform that connects customer feedback to product areas, user segments, releases, support burden, and revenue impact—with every conclusion traceable to the original customer evidence.Initial target userA product manager at a B2B SaaS company.The core questionWhich customer problems deserve further investigation, and what evidence supports them?Notice the wording: further investigation, not “what feature must we build?”The system should organize evidence. It should not pretend the LLM can make the final product decision.3. Example end-to-end workflowSuppose a product manager asks:Why are enterprise customers struggling with onboarding?Step 1: Understand the queryThe system extracts:{  "product_area": "onboarding",  "customer_segment": "enterprise",  "question_type": "root_cause_analysis",  "time_range": "all_time"}Step 2: Query structured dataSQL calculates:Number of relevant ticketsNumber of affected customersAssociated account valueFrequency by monthDistribution by channelRelevant releasesAverage ticket-resolution timeStep 3: Retrieve qualitative evidenceRAG retrieves:Support-ticket descriptionsRelevant call-transcript segmentsChurn commentsProduct reviewsInterview excerptsStep 4: Group the feedbackThe system finds several patterns:SSO configuration is confusing.Users cannot bulk-import team members.Role permissions are unclear.Setup documentation is outdated.Step 5: Generate an evidence reportOpportunity: Improve enterprise onboardingPrimary affected segment:Enterprise account administratorsMain problems:1. SSO configuration requires repeated support involvement.2. Users must invite team members individually.3. Permission roles are difficult to understand.Evidence:- 39 support tickets- 14 sales conversations- 7 churn responses- 4 delayed opportunitiesEstimated support burden:Approximately 31 support hours during the selected period.Counterevidence:- The issue appeared in only 2% of SMB feedback.- Customers using the onboarding service rarely reported the problem.Possible product responses:- Guided SSO setup- CSV user import- Permission-role templates- Updated onboarding documentationThese are possible responses, not automatically approved recommendations.The PM can click every count or claim and inspect the underlying evidence.4. The most important concept: feedback atomsOne support ticket or review can contain several independent observations.Example review:The interface looks great, but CSV export crashes for large files. Support fixed my account quickly, although I still cannot schedule exports.This contains at least four separate signals:Positive feedback about interface designBug involving large CSV exportsPositive support experienceFeature request for scheduled exportsIf you embed the entire review as one chunk, retrieval becomes noisy.Instead, split it into feedback atoms.[  {    "statement": "The interface looks great.",    "type": "praise",    "product_area": "user_interface",    "sentiment": "positive"  },  {    "statement": "CSV export crashes for large files.",    "type": "bug",    "product_area": "exports",    "sentiment": "negative",    "severity": "high"  },  {    "statement": "Support fixed my account quickly.",    "type": "service_feedback",    "product_area": "customer_support",    "sentiment": "positive"  },  {    "statement": "I cannot schedule exports.",    "type": "feature_request",    "product_area": "exports",    "sentiment": "negative"  }]This is more useful than ordinary 500-token chunking.A Python representationfrom datetime import datetimefrom typing import Literalfrom pydantic import BaseModel, Fieldclass FeedbackAtom(BaseModel):    atom_id: str    feedback_id: str    statement: str = Field(min_length=3)    feedback_type: Literal[        "bug",        "feature_request",        "usability_problem",        "praise",        "question",        "service_feedback",        "other",    ]    product_area: str    sentiment: Literal["positive", "neutral", "negative", "mixed"]    severity: Literal["low", "medium", "high", "critical"]    customer_segment: str | None = None    product_version: str | None = None    occurred_at: datetime | None = None    extraction_confidence: float = Field(ge=0.0, le=1.0)What this code doesBaseModel comes from Pydantic. It validates each generated feedback atom before you store it.For example:extraction_confidence: float = Field(ge=0.0, le=1.0)means the confidence must be between 0 and 1. Invalid output is rejected instead of silently entering your database.Literal restricts a value to a fixed set of categories. The model cannot randomly produce categories such as "extremely_bad_bug" unless your schema permits it.5.
